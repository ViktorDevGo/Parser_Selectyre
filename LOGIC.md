# Логика обновления таблиц Selectyre

## Обзор

Система использует **UPSERT** (UPDATE or INSERT) логику для обновления данных о шинах и дисках. Это означает, что данные **никогда не удаляются**, а только добавляются или обновляются.

## Основные принципы

### 1. Уникальный идентификатор
- Каждая запись имеет уникальный `id` (PRIMARY KEY)
- `id` используется для определения: существует запись или это новая

### 2. Временные метки
- `created_at` - дата первого появления записи в БД (устанавливается один раз)
- `updated_at` - дата последнего обновления записи (обновляется при каждом изменении)

### 3. Источник данных
- CSV файлы скачиваются с серверов Selectyre каждые 4 часа
- Кодировка: `windows-1251`
- Разделитель: `;`

## Процесс обновления (пошагово)

```
┌─────────────────────────────────────────────────────────────────┐
│ ЭТАП 1: СКАЧИВАНИЕ CSV                                          │
└─────────────────────────────────────────────────────────────────┘

1. Скачивается новый CSV файл из URL
2. Сохраняется во временный файл (.new)
3. Проверяется размер файла (> 1000 байт)
4. Если ошибка - процесс останавливается

┌─────────────────────────────────────────────────────────────────┐
│ ЭТАП 2: ПАРСИНГ CSV                                             │
└─────────────────────────────────────────────────────────────────┘

1. Открывается CSV с кодировкой windows-1251
2. Читается заголовок (первая строка)
3. Каждая строка парсится в структуру данных
4. Числовые поля конвертируются (quantity → int, price → float)

┌─────────────────────────────────────────────────────────────────┐
│ ЭТАП 3: UPSERT В БАЗУ ДАННЫХ                                    │
└─────────────────────────────────────────────────────────────────┘

Для каждой записи из CSV:

   ┌──────────────────┐
   │ Проверка ID      │
   └────────┬─────────┘
            │
    ┌───────┴────────┐
    │                │
   ДА               НЕТ
    │                │
    ▼                ▼
┌────────┐      ┌─────────┐
│ UPDATE │      │ INSERT  │
└────────┘      └─────────┘

┌─────────────────────────────────────────────────────────────────┐
│ ЭТАП 4: ЛОГИРОВАНИЕ                                             │
└─────────────────────────────────────────────────────────────────┘

1. Записывается статистика:
   - Сколько записей обработано
   - Сколько новых добавлено
   - Сколько обновлено
   - Время выполнения

2. Сохраняется в лог-файл с датой
```

## Детальная логика UPSERT

### SQL запрос (упрощенно)

```sql
INSERT INTO таблица (id, поле1, поле2, ..., created_at, updated_at)
VALUES ($1, $2, $3, ..., CURRENT_TIMESTAMP, CURRENT_TIMESTAMP)

ON CONFLICT (id) DO UPDATE SET
    поле1 = EXCLUDED.поле1,
    поле2 = EXCLUDED.поле2,
    ...
    updated_at = CURRENT_TIMESTAMP;
```

### Что происходит

#### Случай 1: ID не существует (новая запись)
```
CSV: ID=t999999, brand=NewBrand, price=5000

БД до:  (пусто)

SQL выполняет: INSERT
  id = t999999
  brand = NewBrand
  price = 5000
  created_at = 2026-03-26 15:00:00
  updated_at = 2026-03-26 15:00:00

БД после:
  id: t999999
  brand: NewBrand
  price: 5000
  created_at: 2026-03-26 15:00:00
  updated_at: 2026-03-26 15:00:00
```

#### Случай 2: ID существует (обновление)
```
CSV: ID=t123456, brand=Pirelli, price=15000, quantity=10

БД до:
  id: t123456
  brand: Pirelli
  price: 12000
  quantity: 5
  created_at: 2026-03-20 10:00:00
  updated_at: 2026-03-25 14:00:00

SQL выполняет: UPDATE
  brand = Pirelli (без изменений)
  price = 15000 (изменено!)
  quantity = 10 (изменено!)
  updated_at = 2026-03-26 15:00:00 (обновлено!)
  created_at остается 2026-03-20 10:00:00 (НЕ меняется!)

БД после:
  id: t123456
  brand: Pirelli
  price: 15000 ← изменилось
  quantity: 10 ← изменилось
  created_at: 2026-03-20 10:00:00 ← НЕ изменилось
  updated_at: 2026-03-26 15:00:00 ← обновилось
```

#### Случай 3: ID существует, но данные не изменились
```
CSV: ID=t555555, brand=Yokohama, price=8000

БД до:
  id: t555555
  brand: Yokohama
  price: 8000
  created_at: 2026-03-15 12:00:00
  updated_at: 2026-03-25 16:00:00

SQL выполняет: UPDATE (все равно обновляет!)
  brand = Yokohama (без изменений)
  price = 8000 (без изменений)
  updated_at = 2026-03-26 15:00:00 (обновлено!)
  created_at остается 2026-03-15 12:00:00

БД после:
  id: t555555
  brand: Yokohama
  price: 8000
  created_at: 2026-03-15 12:00:00 ← НЕ изменилось
  updated_at: 2026-03-26 15:00:00 ← обновилось (даже если данные те же!)
```

## Обработка пакетами

Для оптимизации производительности данные вставляются пакетами:

```python
batch_size = 500  # записей за раз

for row in csv_reader:
    batch_data.append(row)

    if len(batch_data) >= batch_size:
        execute_batch(cursor, upsert_query, batch_data)
        conn.commit()
        batch_data = []

# Оставшиеся записи
if batch_data:
    execute_batch(cursor, upsert_query, batch_data)
    conn.commit()
```

**Преимущества:**
- ✅ Быстрее чем по одной записи
- ✅ Меньше нагрузки на БД
- ✅ Меньше сетевых запросов
- ✅ Можно отслеживать прогресс

## Статистика обновления

После каждого обновления вычисляется:

### 1. Общее количество
```sql
SELECT COUNT(*) FROM таблица;
```

### 2. Новые записи (созданные сегодня)
```sql
SELECT COUNT(*)
FROM таблица
WHERE DATE(created_at) = CURRENT_DATE;
```

### 3. Обновленные записи (обновленные сегодня, но не созданные)
```sql
SELECT COUNT(*)
FROM таблица
WHERE DATE(updated_at) = CURRENT_DATE
  AND DATE(created_at) != CURRENT_DATE;
```

### 4. Без изменений
```sql
total_records - updated_today
```

## Важные моменты

### ✅ ЧТО ПРОИСХОДИТ

1. **Новые товары добавляются**
   - Появился новый диск → добавится в БД

2. **Существующие товары обновляются**
   - Изменилась цена → обновится
   - Изменился остаток → обновится
   - Изменилось название → обновится

3. **updated_at всегда обновляется**
   - Даже если данные не изменились
   - Это позволяет отследить когда товар последний раз был в прайсе

4. **created_at никогда не меняется**
   - Сохраняет дату первого появления товара

### ❌ ЧТО НЕ ПРОИСХОДИТ

1. **Данные НЕ удаляются**
   - Если товар исчез из CSV, он остается в БД
   - Просто его updated_at перестанет обновляться

2. **История изменений НЕ сохраняется**
   - Если цена была 5000, потом стала 6000, потом 7000
   - В БД будет только последнее значение: 7000
   - История 5000 → 6000 → 7000 не сохраняется

3. **Дубликаты НЕ создаются**
   - Один ID = одна запись в БД
   - Невозможно иметь два товара с одинаковым ID

## Обработка ошибок

### 1. Ошибка скачивания CSV
```python
if download_failed:
    logger.error("Не удалось скачать CSV")
    return 1  # выход с ошибкой
    # БД остается в прежнем состоянии
```

### 2. Ошибка подключения к БД
```python
try:
    conn = psycopg2.connect(...)
except Exception as e:
    logger.error(f"Ошибка подключения: {e}")
    return 1
```

### 3. Ошибка парсинга строки
```python
try:
    quantity = int(row['QUANTITY'])
except:
    logger.warning(f"Ошибка в строке {row_num}")
    continue  # пропускаем эту строку, продолжаем дальше
```

### 4. Ошибка при вставке пакета
```python
try:
    execute_batch(cursor, query, batch_data)
    conn.commit()
except Exception as e:
    logger.error(f"Ошибка вставки: {e}")
    conn.rollback()  # откатываем транзакцию
    raise  # останавливаем процесс
```

## Транзакционность

### Пакетная вставка с коммитами
```python
for i, row in enumerate(csv_reader):
    batch_data.append(row)

    if len(batch_data) >= 500:
        try:
            execute_batch(cursor, upsert_query, batch_data)
            conn.commit()  # ✅ сохраняем этот пакет
        except:
            conn.rollback()  # ❌ откатываем только этот пакет
            raise
        batch_data = []
```

**Что это значит:**
- Если обновление падает на 1500-й записи
- Первые 1000 записей уже сохранены (2 пакета по 500)
- Откатывается только третий пакет (записи 1001-1500)
- При следующем запуске:
  - Записи 1-1000 будут обновлены (updated_at изменится)
  - Записи 1001-конец будут вставлены/обновлены заново

## Производительность

### Текущие показатели

**Шины (6,829 записей):**
- Скачивание CSV: ~2-3 сек
- Обработка и UPSERT: ~23-24 сек
- **Общее время: ~26 сек**

**Диски (3,005 записей):**
- Скачивание CSV: ~2 сек
- Обработка и UPSERT: ~12 сек
- **Общее время: ~14 сек**

**Всего (9,834 записи):**
- **~40 секунд** на полное обновление

### Оптимизации

1. **Пакетная вставка** (500 записей)
   - Вместо 6829 запросов → 14 запросов
   - Ускорение в ~30-50 раз

2. **Индексы на часто используемых полях**
   - brand, model, diameter, pcd
   - Ускорение поиска и UPDATE

3. **Подготовленные запросы** (execute_batch)
   - Парсинг SQL происходит один раз
   - Переиспользование плана выполнения

## Мониторинг изменений

### Как узнать что изменилось?

#### 1. Товары добавленные сегодня
```sql
SELECT * FROM Selectyre_tyer
WHERE DATE(created_at) = CURRENT_DATE;
```

#### 2. Товары обновленные сегодня
```sql
SELECT * FROM Selectyre_tyer
WHERE DATE(updated_at) = CURRENT_DATE
  AND DATE(created_at) != CURRENT_DATE;
```

#### 3. Товары не обновлявшиеся более 7 дней
```sql
SELECT * FROM Selectyre_tyer
WHERE updated_at < NOW() - INTERVAL '7 days';
```
*Вероятно, эти товары больше не в прайсе*

#### 4. Разница между created_at и updated_at
```sql
SELECT
    id,
    brand,
    model,
    created_at,
    updated_at,
    (updated_at - created_at) as lifetime
FROM Selectyre_tyer
WHERE created_at != updated_at
ORDER BY lifetime DESC;
```
*Показывает как долго товар в прайсе и обновляется*

## Расширение системы

### Добавление истории изменений

Если нужно отслеживать изменения цен:

```sql
CREATE TABLE price_history (
    id SERIAL PRIMARY KEY,
    product_id TEXT,
    product_type TEXT, -- 'tire' or 'rim'
    old_price NUMERIC(10,2),
    new_price NUMERIC(10,2),
    changed_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

Логика в Python:
```python
# Перед UPDATE проверяем старую цену
cursor.execute("SELECT price FROM Selectyre_tyer WHERE id = %s", (id,))
old_price = cursor.fetchone()

# После UPDATE сохраняем историю
if old_price and old_price[0] != new_price:
    cursor.execute("""
        INSERT INTO price_history (product_id, product_type, old_price, new_price)
        VALUES (%s, %s, %s, %s)
    """, (id, 'tire', old_price[0], new_price))
```

### Добавление soft delete

Если нужно помечать удаленные товары:

```sql
ALTER TABLE Selectyre_tyer ADD COLUMN is_active BOOLEAN DEFAULT TRUE;
```

Логика:
```python
# Перед обновлением помечаем все как неактивные
cursor.execute("UPDATE Selectyre_tyer SET is_active = FALSE")

# При UPSERT помечаем как активные
upsert_query += ", is_active = TRUE"
```

Товары которых нет в новом CSV останутся с `is_active = FALSE`

## Заключение

### Преимущества текущей логики

✅ **Простота** - понятная логика INSERT/UPDATE
✅ **Безопасность** - данные не удаляются
✅ **Производительность** - пакетная обработка
✅ **Отслеживаемость** - created_at/updated_at метки

❌ Нет истории изменений цен
❌ Нет определения удаленных товаров
